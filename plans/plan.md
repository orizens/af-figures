# Plan: Appfigures Reviews Webapp

Build a pure client-side React SPA using **Vite** as the dev/build tool and **Bun** as the package manager and test runner, with TanStack Router, TanStack Query, and Tailwind v4. The app fetches data directly from the external API ‚Äî no backend, no SSR. URL is the sole app state ‚Äî deep-linkable with validated search params. Browser back/forward navigation is fully supported.

---

## Tech Stack

- **Vite** ‚Äî dev server (HMR) + production build (CSR, static output)
- **Bun** ‚Äî package manager + unit test runner (`bun test`)
- React 19 + TypeScript (strict)
- TanStack Router v1 (URL as state, search param validation, browser history)
- TanStack Query v5 (`useInfiniteQuery` for load more)
- Tailwind CSS v4 via `@tailwindcss/vite` plugin
- Zod ‚Äî search param schema + validation
- Playwright ‚Äî end-to-end test coverage

## Coding Guidelines

### General

- **`const` always** ‚Äî never use `let` unless reassignment is genuinely required; never use `var`
- **No deep nesting** ‚Äî max 2 levels of conditional/loop nesting inside a function body; extract early-return guards or helper functions instead
- **Encapsulation** ‚Äî each module exposes only what consumers need; internals are not exported
- **Readability over cleverness** ‚Äî prefer explicit, named intermediary variables over chained one-liners
- **Pure functions** ‚Äî utility/business logic functions must be side-effect free and independently testable (e.g. `groupReviewsByDate`)
- **TypeScript strict** ‚Äî no `any`; all function parameters and return types explicitly typed; use Zod for runtime validation at system boundaries (URL params, API responses)

### Architecture

- **Feature-based modules** ‚Äî domain code (components, hooks, utils) lives inside `src/features/<feature>/`; a feature module owns everything specific to that domain
- **Shared layer** ‚Äî cross-feature infrastructure (`api/`, `hooks/`, `utils/`, `types/`) lives in `src/shared/`; features import from shared, never from each other
- **App shell** ‚Äî `src/components/` holds only app-level layout components (`AppHeader`, `ErrorBoundary`) that are not tied to a specific feature
- **Scalability** ‚Äî adding a new feature means creating a new directory under `src/features/`; zero changes required to shared or shell layers
- **Business logic in hooks or pure functions** ‚Äî components should contain only rendering logic; data fetching, filtering, transformation, and formatting live in custom hooks or utility functions
- **No barrel files** ‚Äî do not create `index.ts` re-export files; import directly from the source module (e.g. `@/features/reviews/components/ReviewCard`) to avoid circular dependency risks and keep tree-shaking reliable
- **Single responsibility** ‚Äî each file does one thing; if a file needs a comment to explain what it does, split it

### React

- **No prop drilling past 2 levels** ‚Äî use TanStack Router's `useSearch` / `useNavigate` instead of passing search state as props
- **Custom hooks for side effects** ‚Äî `useDebounce`, `useReviews`, etc. keep components declarative
- **Error boundaries at feature boundaries** ‚Äî `ReviewsList` has its own `ErrorBoundary`; the app root has a top-level `AppErrorBoundary`
- **Loading states are explicit** ‚Äî never render partial data without a loading indicator

### Accessibility

- All interactive elements must have accessible labels (`aria-label` or visible text)
- Search controls are wrapped in `<search aria-label="Reviews search">` (the native HTML landmark element); falls back to `<form role="search" aria-label="Reviews search">` if SSR compatibility is a concern
- Search input: `<input type="search" aria-label="Search reviews" />` ‚Äî `type="search"` gives the implicit `searchbox` role; do NOT add `role="search"` to the input itself
- Rating dropdown: `aria-label="Filter by rating"`, each option has descriptive text ("1 star", "2 stars", etc.)
- Review cards: `role="article"`, `aria-label` derived from title + author
- Review groups (date sections): wrapped in `<section>` with `aria-labelledby` pointing to the heading
- Load More button: `aria-label="Load more reviews"`, `aria-busy` reflects loading state
- Star rating display: `role="img"` with `aria-label="X out of 5 stars"`
- Loading spinner: `role="status"`, `aria-live="polite"`, `aria-label="Loading reviews"`
- Error message regions: `role="alert"`, `aria-live="assertive"`
- All ARIA roles and labels must be stable identifiers ‚Äî Playwright tests target them directly via `getByRole` and `getByLabel`

### URL & Navigation

- **URL is the only app state** ‚Äî `q`, `rating`, `page` are the single source of truth; no local state duplicates them except the debounced search input buffer
- **Browser back/forward fully supported** ‚Äî TanStack Router uses `createBrowserHistory` by default; every `navigate()` call pushes a real browser history entry so the back button re-renders the app to the previous filter state and re-fetches accordingly
- **Deep links work** ‚Äî opening `/?q=crash&rating=1&page=2` or `/?q=love&rating=4,5` loads the correct data; `rating` is comma-separated in the URL (human-readable) and coerced to `number[]` by the Zod `validateSearch` schema
- **Invalid params are silently dropped** ‚Äî Zod `validateSearch` coerces or strips bad values to safe defaults

## Testing Strategy (TDD / Spec-Driven)

### Approach

- **Write specs first** ‚Äî before implementing a feature, write the Playwright spec describing the expected user behaviour; the spec acts as the requirement. Implementation is done to make the spec pass.
- **Unit tests for pure logic** ‚Äî `groupReviewsByDate` and any shared utility must have `*.test.ts` files run with `bun test` covering all section boundaries (today, yesterday, this week, etc.) including edge cases (empty input, review on a Monday, last day of month)
- **E2E tests for user scenarios** ‚Äî Playwright specs cover full browser flows including URL state, back/forward navigation, and ARIA-accessible selectors

### Playwright Specs ‚Äî `tests/e2e/reviews.spec.ts`

Full scenario coverage for the reviews feature:

| Scenario                   | Description                                                                             |
| -------------------------- | --------------------------------------------------------------------------------------- |
| Default load               | Visits `/`, sees "All Reviews", list renders with grouped sections                      |
| Search keyword             | Types "love" in search, waits debounce, URL updates to `?q=love`, list refreshes        |
| Filter by rating           | Selects "5 stars" from dropdown, URL updates to `?rating=5`, list refreshes immediately |
| Combined filters           | Types "crash" + selects "1 star", URL = `?q=crash&rating=1`, correct results shown      |
| Deep link                  | Navigates directly to `/?q=love&rating=4,5`, filters pre-populate, data loads           |
| Invalid params             | Navigates to `/?rating=99&page=abc`, app loads with defaults, no error thrown           |
| Load more                  | Clicks "Load More", page 2 loads and appends, URL updates to `page=2`                   |
| Refresh with page          | With `?page=2`, refreshes browser, same 50 results visible                              |
| Error state                | API returns 500, error message is shown in list area, app header still visible          |
| No results                 | Search returns 0 results, empty state message is shown                                  |
| Summary line (no filter)   | No filters active ‚Üí summary shows "All Reviews"                                         |
| Summary line (with filter) | Filter active ‚Üí summary shows "Showing X reviews"                                       |

### Playwright Specs ‚Äî `tests/e2e/navigation.spec.ts`

Browser history navigation scenarios:

| Scenario                 | Description                                                                        |
| ------------------------ | ---------------------------------------------------------------------------------- |
| Back after search        | Types "love" (URL ‚Üí `?q=love`), then presses back, URL reverts to `/`, list resets |
| Back after rating filter | Selects rating 5 (URL ‚Üí `?rating=5`), presses back, URL reverts, list resets       |
| Forward after back       | Back then forward re-applies filter, list re-fetches                               |
| Multi-step back          | search + rating + load more = 3 history entries; pressing back 3√ó returns to `/`   |

### Selector Strategy

- Prefer `getByRole` and `getByLabel` (ARIA-based) over CSS selectors ‚Äî this enforces accessibility requirements and makes tests resilient to style changes
- Use `data-testid` only as a last resort for elements without semantic roles
- Example: `page.getByRole('searchbox', { name: 'Search reviews' })` not `page.locator('input[type=text]')`

## Key Decisions

- **URL is source of truth**: `q`, `rating`, `page` search params; no Redux/Zustand needed
- **Browser history**: TanStack Router pushes real history entries on every `navigate()` ‚Äî back/forward work natively; navigating back re-renders the component with previous search params and triggers a query refetch via the changed `queryKey`
- **Debounce `q`**: local input state (controlled input) initialized from `useSearch().q` ‚Üí 400ms debounce via `useDebounce` ‚Üí `navigate`; the URL lags intentionally to avoid API spam. A `useEffect(() => { setLocalQ(urlQ) }, [urlQ])` syncs the input back when URL changes externally (back/forward) ‚Äî idempotent when the URL change was caused by our own navigate.
- **`rating`**: immediate `navigate` on dropdown change
- **`page`**: incremented on Load More, reset to 1 on any filter change
- **`useInfiniteQuery` over `useSuspenseInfiniteQuery`**: Suspense flashes the loading spinner on every filter change; `useInfiniteQuery` with `placeholderData: (prev) => prev` keeps old results visible while the new query loads, then swaps in when ready
- **`useInfiniteQuery` page restoration**: `initialPageParam=1`; on first mount with `urlPage > 1`, the `queryFn` requests `count=urlPage*25` ‚Äî single network request returning all accumulated reviews; no `useEffect` waterfall
- **Zod `validateSearch`**: invalid params (e.g. `rating=99`) are silently dropped; safe defaults always applied
- **Date grouping**: pure function in `features/reviews/utils/` ‚Äî called inside `select` of `useInfiniteQuery`; easily unit-tested with injected `now` parameter
- **Two error boundaries**: `AppErrorBoundary` (root) + `ReviewsErrorBoundary` (list only)
- **Tailwind theme**: CSS custom properties in `:root` ‚Äî colors and radius customisable without touching component classes
- **Accessibility drives selectors**: ARIA roles/labels are required for Playwright tests to work ‚Äî this enforces accessibility as a side effect of the testing strategy

## Further Considerations

1. **API response shape** ‚Äî Fetch `https://appfigures.com/_u/careers/api/reviews` and inspect the JSON before finalising `src/shared/types/reviews.ts`. Confirm field names for `total`, `app`, and `Review` properties.
2. **Page restoration on refresh** ‚Äî The sequential `fetchNextPage` loop fires N API calls if `page=N`. If the API supports `count=N*25`, a single `useQuery` call at mount avoids multiple round trips. Confirm API limits.
3. **Rating dropdown UX** ‚Äî The example shows `rating=4,5` (multi-value), so multi-select checkboxes are assumed. Confirm with design whether single-select ("1 star", "2 stars" exclusively) is preferred.
4. **Playwright config** ‚Äî Set `baseURL: 'http://localhost:4200'` in `playwright.config.ts`; add a `webServer: { command: 'bun run dev', port: 4200, reuseExistingServer: true }` to auto-start Vite before running tests.
5. **Static deployment** ‚Äî `bun run build` produces a `dist/` folder of static files; deploy to any CDN/static host (Netlify, Vercel, GitHub Pages). No server runtime required.
